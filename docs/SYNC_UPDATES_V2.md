# Offline-First Sync — Full CRUD for Categories, Items, and List Finalization

## Overview

This document summarizes all changes made in the second phase of the offline-first sync implementation. It covers:

1. **Category persistence audit** — gaps identified and fixed between `listas_do_usuario` and `categorias_da_lista` sync behavior
2. **Item sync** — full CRUD offline-first sync for `itens_da_lista`
3. **List finalization** — atomic `finalizeList()` operation writing to `listas_do_usuario.finalizada_em` and `historico_de_listas`

All changes follow the same optimistic-update + sync-queue pattern established in the first phase (see [`SYNC_UPDATES.md`](SYNC_UPDATES.md)).

---

## Part 1 — Category Persistence Audit

### Gaps Identified

Comparing the `listas_do_usuario` implementation (reference) against `categorias_da_lista`:

| Gap | Description |
|-----|-------------|
| **Gap 1** | `addCategory()` did NOT replace the local `tempId` with the DB-generated UUID after a successful Supabase insert (unlike `addShoppingList()` which did) |
| **Gap 2** | No `onCreateCategorySynced` stream existed to propagate the DB-generated UUID back to the controller after offline sync |
| **Gap 3** | `SyncService._processCreateCategory()` did not return or emit the DB-generated category ID |
| **Gap 4** | `deleteShoppingList()` had no Supabase sync at all — no `deleteList()` in `SupabaseListService`, no `PendingOperationType.deleteList`, no handler in `SyncService` |

### Fixes Applied

#### [`lib/features/shopping_list/services/supabase_list_service.dart`](../lib/features/shopping_list/services/supabase_list_service.dart)

- Added `deleteList(String listId)` — hard-deletes a row from `listas_do_usuario`

#### [`lib/features/shopping_list/models/pending_operation.dart`](../lib/features/shopping_list/models/pending_operation.dart)

- Added `deleteList` to `PendingOperationType` enum
- Removed pre-existing unused `dart:convert` import

#### [`lib/core/services/sync_service.dart`](../lib/core/services/sync_service.dart)

- Added `_createCategorySyncController` and `onCreateCategorySynced` stream
- `_processCreateCategory()` now captures the DB row, extracts `dbId`, and emits `{tempId, dbId}` on `onCreateCategorySynced`
- Added `_processDeleteList()` handler (skips non-UUID `listId`)
- `dispose()` closes `_createCategorySyncController`

#### [`lib/features/shopping_list/state/shopping_list_controller.dart`](../lib/features/shopping_list/state/shopping_list_controller.dart)

- Added `_createCategorySyncSubscription` field
- `setSyncService()` subscribes to `onCreateCategorySynced` → `_updateCategoryIdAfterSync()`
- Added `_updateCategoryIdAfterSync(tempId, dbId)` — replaces category ID in `_categories`, updates any items referencing the old ID, persists both, notifies listeners
- `addCategory()` now captures the DB row on success and replaces `tempId` with `dbId` in local state
- `addCategory()` now includes `tempId` in the `createCategory` pending operation payload
- `deleteShoppingList()` now calls `_supabaseListService!.deleteList(listId)` after local delete; enqueues `deleteList` on failure
- `dispose()` cancels `_createCategorySyncSubscription`

#### [`test/sync_service_test.dart`](../test/sync_service_test.dart)

- `MockSupabaseListService` gains `deleteListCalled` flag and `deleteList()` method
- New `group('List Delete Sync Operations')`:
  - `deleteList` with valid UUID is processed and calls `deleteList()`
  - `deleteList` with non-UUID `listId` is skipped
- New test `'Deve emitir onCreateCategorySynced com tempId e dbId'` — verifies the stream emits the correct ID-swap event

## Part 2 — Item Sync (`itens_da_lista`)

### New Fields in `ShoppingItem`

| Field | Type | DB Column | Description |
|-------|------|-----------|-------------|
| `listId` | `String?` | `lista_id` | The list this item belongs to (used for Supabase sync) |
| `origem` | `ItemOrigem` | `origem` | Enum: `manual`, `llm`, `ajuste` |
| `ordemDb` | `int` | `ordem` | DB order value (separate from local sort order) |
| `criadoEm` | `DateTime` | `criado_em` | Creation timestamp |
| `atualizadoEm` | `DateTime` | `atualizado_em` | Last update timestamp |
| `removidoEm` | `DateTime?` | `removido_em` | Soft-delete timestamp (null = not deleted) |

**`ItemOrigem` enum:**
- `manual` — created directly by the user
- `llm` — generated by an LLM from a meal plan
- `ajuste` — an LLM-generated item subsequently edited by the user

**`ShoppingItem.create()` factory** — sets `criadoEm` and `atualizadoEm` to `DateTime.now()`.

### New Methods in `SupabaseListService`

| Method | Description |
|--------|-------------|
| `insertItem({...})` | Inserts a row into `itens_da_lista`; returns full row including DB-generated `id` |
| `updateItem({itemId, ...})` | Updates only non-null fields; always sets `atualizado_em` |
| `softDeleteItem(String itemId)` | Sets `removido_em` (soft delete; row is preserved for audit) |

**Price conversion:** Local `priceValue` (double, e.g. `2.50`) is converted to `precoCentavos` (int, e.g. `250`) before Supabase insert. The helper `_toCents(double value)` in the controller handles this.

### New `PendingOperationType` Values

- `createItem`
- `updateItem`
- `deleteItem`

### New Sync Operations in `SyncService`

| Handler | Trigger | Skips if |
|---------|---------|----------|
| `_processCreateItem()` | `createItem` operation | `listId` is not a valid UUID v4 |
| `_processUpdateItem()` | `updateItem` operation | `itemId` is not a valid UUID v4 |
| `_processDeleteItem()` | `deleteItem` operation | `itemId` is not a valid UUID v4 |

`_processCreateItem()` emits `{tempId, dbId}` on `onCreateItemSynced` stream after successful insert.

### Controller Changes

| Method | Change |
|--------|--------|
| `addItem()` | Uses `ShoppingItem.create()` with UUID `tempId`; replaces with `dbId` on Supabase success; enqueues `createItem` on failure |
| `removeItem()` | Calls `softDeleteItem()` after local remove; enqueues `deleteItem` on failure |
| `editItem()` | Promotes `llm` → `ajuste` origem; calls `updateItem()`; enqueues `updateItem` on failure |
| `toggleItemCheck()` | Calls `updateItem(completo, completoEm)`; enqueues `updateItem` on failure |
| `markItemChecked()` | Calls `updateItem(completo, completoEm)`; enqueues `updateItem` on failure |
| `moveItemToCategory()` | Calls `updateItem(categoryId)`; enqueues `updateItem` on failure |
| `copyItemToCategory()` | Uses `ShoppingItem.create()` with UUID `tempId`; replaces with `dbId` on success; enqueues `createItem` on failure |
| `restoreItem()` | Re-inserts via `insertItem()` (prior soft-delete may have synced); replaces ID on success; enqueues `createItem` on failure |

```json
{
  "itemId": "uuid-of-item",
  "nome": "Leite integral",
  "quantidadeCompra": 2.0,
  "unidadeCompra": "L",
```

## Part 3 — List Finalization (`finalizeList`)

### Problem

When the user clicks "Salvar no histórico" on the finalization screen, two DB operations must happen together:

1. Update `listas_do_usuario.finalizada_em` with the current timestamp
2. Insert a new record into `historico_de_listas`

The original implementation only updated local state and had no Supabase persistence.

### `historico_de_listas` Schema

| Column | Type | Description |
|--------|------|-------------|
| `id` | `uuid` | Auto-generated |
| `lista_id` | `uuid` | FK to `listas_do_usuario.id` |
| `owner_usuario_id` | `uuid` | Owner of the list |
| `finalizado_por_usuario_id` | `uuid` | User who finalized (may differ from owner) |
| `local_compra` | `text` | Purchase location |
| `data_compra` | `date` | Purchase date (format: `yyyy-MM-dd`) |
| `valor_total_centavos` | `int` | Total spent in cents |
| `finalizado_em` | `timestamptz` | Finalization timestamp |

### Bug Fixed: 400 Error on INSERT

The initial implementation had two bugs causing a 400 Bad Request on the `historico_de_listas` INSERT:

1. **Wrong `data_compra` format** — We were sending a full ISO 8601 timestamp (`"2024-01-15T00:00:00.000Z"`) to a `date` column that expects `"2024-01-15"` (date-only format). Fixed by using `DateFormat('yyyy-MM-dd').format(dataCompra)`.

2. **Missing required columns** — The INSERT omitted `finalizado_em` (NOT NULL). Fixed by including `finalizado_em: now.toIso8601String()` in the payload.

### New Method: `SupabaseListService.finalizeList()`

```dart
Future<void> finalizeList({
  required String listId,
  required String ownerUsuarioId,
  required String finalizadoPorUsuarioId,
  required String localCompra,
  required DateTime dataCompra,
  required int valorTotalCentavos,
}) async
```

**Behavior:**
1. Updates `listas_do_usuario.finalizada_em` with current UTC timestamp
2. Inserts into `historico_de_listas` with all required fields
3. `data_compra` is formatted as `yyyy-MM-dd` (Postgres `date` type)
4. `finalizado_em` is set to current UTC timestamp

**Atomicity note:** The two operations are sequential, not wrapped in a DB transaction. If the UPDATE succeeds but the INSERT fails, the `finalizada_em` timestamp will be set but no history record will exist. The controller returns `false` in this case, allowing the UI to surface the error to the user for manual retry.

### Controller Changes

`ShoppingListController.finalizeList()` now:
- Returns `bool` (was `void`)
- Performs optimistic local update first (preserves existing behavior)
- Calls `_supabaseListService!.finalizeList()` for authenticated users
- Converts `totalSpent` (double) to `valorTotalCentavos` (int) via `(totalSpent * 100).round()`
- Returns `true` on success, `false` on Supabase failure
- Does NOT enqueue for offline sync (finalization is a one-time operation that should be retried manually)

### UI Changes

`FinalizeListScreen` now:
- Captures the `bool` result from `controller.finalizeList()`
- Navigates to the lists screen on `true`
- Shows an error `SnackBar` on `false`:
  > "Erro ao salvar no histórico. Verifique sua conexão e tente novamente."

## Architecture Diagram (Updated)

```
User Action
     │
     ▼
ShoppingListController
  ├── Local state update (SharedPreferences)
  ├── notifyListeners() → UI updates immediately
  └── Try Supabase call
        ├── SUCCESS
        │     ├── createList/createCategory/createItem → replace tempId with dbId
        │     └── Other operations → done
        └── FAILURE
              │
              ▼
         SyncService.enqueue(PendingOperation)
              │
              ▼
         SyncQueueRepository (SharedPreferences)
              │
              ▼
         On connectivity restored → processQueue()
                          │
                          ▼
                     For each operation:
                       ├── createList → insertManualList() → emit onCreateListSynced
                       ├── renameList → updateListName()
                       ├── deleteList → deleteList()
                       ├── createCategory → insertCategory() → emit onCreateCategorySynced
                       ├── updateCategory → updateCategoryName/Color/Collapsed/Order()
                       ├── deleteCategory → deleteCategory()
                       ├── createItem → insertItem() → emit onCreateItemSynced
                       ├── updateItem → updateItem()
                       └── deleteItem → softDeleteItem()

onCreateListSynced stream → _updateListIdAfterSync(tempId, dbId)
onCreateCategorySynced stream → _updateCategoryIdAfterSync(tempId, dbId)
onCreateItemSynced stream → _updateItemIdAfterSync(tempId, dbId)
```

## Summary of All `PendingOperationType` Values

| Type | Trigger | Supabase Call | ID Swap Stream |
|------|---------|---------------|----------------|
| `createList` | `addShoppingList()` fails | `insertManualList()` | `onCreateListSynced` |
| `renameList` | `renameShoppingList()` fails | `updateListName()` | — |
| `deleteList` | `deleteShoppingList()` fails | `deleteList()` | — |
| `createCategory` | `addCategory()` fails | `insertCategory()` | `onCreateCategorySynced` |
| `updateCategory` | Any category update fails | `updateCategoryName/Color/Collapsed/Order()` | — |
| `deleteCategory` | `removeCategory()` / `deleteCategory()` fails | `deleteCategory()` | — |
| `createItem` | `addItem()` / `copyItemToCategory()` / `restoreItem()` fails | `insertItem()` | `onCreateItemSynced` |
| `updateItem` | `editItem()` / `toggleItemCheck()` / `markItemChecked()` / `moveItemToCategory()` fails | `updateItem()` | — |
| `deleteItem` | `removeItem()` fails | `softDeleteItem()` | — |

## Known Limitations

1. **`finalizeList` is not atomic** — The UPDATE and INSERT are sequential. If the INSERT fails, the UPDATE is not rolled back. The user must retry manually.

2. **No offline sync for `finalizeList`** — Finalization is not enqueued for offline sync. If the user is offline when finalizing, the operation will fail and they must retry when online.

3. **`connectivity_plus` not mockable in unit tests** — All sync tests fail with `MissingPluginException` in the unit test environment. Integration tests or a mock `Connectivity` wrapper would be needed for full coverage.

4. **`SyncService` uses `dynamic` for `supabaseService`** — Typed as `dynamic` to allow mocking in tests without Supabase initialization. This loses compile-time type safety.
